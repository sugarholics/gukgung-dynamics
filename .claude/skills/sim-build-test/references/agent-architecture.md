# 에이전트 아키텍처 제안

## 작업 유형별 에이전트 분류

### 1. 물리 모델 에이전트 (research/plan)
- **용도**: 수식 분석, 파라미터 탐색, 물리적 타당성 검증
- **도구**: Read, Grep, WebSearch (논문/참고자료)
- **스킬**: physics-engine
- **예시 작업**:
  - "brace height가 너무 낮은 원인 분석"
  - "대변형 빔 모델로 교체 가능성 검토"
  - "실제 개량궁 사양과 파라미터 비교"

### 2. 코드 수정 에이전트 (general-purpose)
- **용도**: JSX 코드 편집, 새 기능 구현
- **도구**: Read, Edit, Write, Grep, Bash
- **스킬**: physics-engine + threejs-viz
- **예시 작업**:
  - "화살 패러독스 모델 추가"
  - "시위 탄성 모델 구현"
  - "UI에 에너지 그래프 추가"

### 3. 테스트/검증 에이전트 (general-purpose)
- **용도**: 빌드, 브라우저 테스트, 시각적 검증
- **도구**: Bash, Chrome tools (navigate, screenshot, javascript_tool)
- **스킬**: sim-build-test
- **예시 작업**:
  - "빌드하고 50% draw에서 스크린샷"
  - "발시 시뮬레이션 실행해서 진동 확인"
  - "모든 슬라이더 극값에서 오류 없는지 확인"

## 병렬 작업 패턴

### 패턴 A: 독립 기능 병렬 개발
```
사용자: "화살 패러독스 추가하고, UI에 에너지 그래프도 넣어줘"

→ Agent 1 (worktree): 화살 패러독스 물리 모델 구현
→ Agent 2 (worktree): 에너지 그래프 UI 구현
→ 결과 병합 후 빌드 & 테스트
```

### 패턴 B: 조사 + 구현 병렬
```
사용자: "대변형 빔 모델로 개선해줘"

→ Agent 1 (Explore): 현재 빔 모델 코드 분석 + 수정 필요 부분 파악
→ Agent 2 (WebSearch): 대변형 빔 이론 참고자료 조사
→ 결과 취합 후 구현 계획 수립
→ Agent 3: 코드 수정
→ Agent 4: 빌드 & 검증
```

### 패턴 C: 수정 후 자동 검증
```
사용자: "탄성계수를 수정해서 만작 15kgf 되게 해줘"

→ 물리 분석 (어떤 EI가 필요한지 계산)
→ 코드 수정 (Edit)
→ 빌드 (Bash: node build_html.js)
→ 브라우저 테스트 (Chrome tools)
→ 결과 보고 + 필요시 반복
```

## worktree 격리 활용

2000줄 JSX 파일의 서로 다른 부분을 동시에 수정하면 충돌이 발생한다.
`isolation: "worktree"`를 사용하면 각 에이전트가 별도 Git 브랜치에서 작업하므로 안전하다.

단, 병합(merge) 시 같은 함수를 수정한 경우 수동 충돌 해결이 필요할 수 있다.
이를 피하려면: 서로 다른 함수/섹션을 담당하도록 작업을 분할할 것.
